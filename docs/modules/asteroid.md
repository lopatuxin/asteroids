# Модуль `asteroid`

## Назначение

Модуль определяет класс `Asteroid extends Entity` — одну из центральных игровых сущностей: летящий, вращающийся, «шершавый» астероид одного из трёх размеров (`large` / `medium` / `small`). Без него нет ни основного объекта стрельбы, ни механики дробления на более мелкие осколки, ни наполнения волны — то есть не существует самой игры. Модуль инкапсулирует всё, что отличает астероид от прочих сущностей: случайно-сгенерированную форму полигона, вращение и правила расщепления при уничтожении.

## Ответственности

- Объявление класса `Asteroid extends Entity` с полями размера, формы, угла поворота и угловой скорости.
- Генерация «шершавой» формы астероида при создании — массив радиусов по равномерно распределённым углам, слегка зашумлённых относительно базового радиуса (один раз, в конструкторе).
- Инициализация собственных физических параметров из `config.ASTEROID` по размеру: радиус коллизии, диапазон линейной скорости, диапазон угловой скорости, амплитуда неровности, число вершин, очки.
- Шаг симуляции: накопление угла поворота (`rotation += angularVelocity * dt`) и делегирование движения с wrap-around через `Entity.integrate(dt)`.
- Отрисовка замкнутой ломаной по массиву вершин `shape` с учётом текущего `rotation` и `position` — белыми линиями, в духе оригинала.
- Реализация `split()` — правила дробления: `large → 2 × medium`, `medium → 2 × small`, `small → []`. Новым осколкам задаётся позиция исходного астероида и два разнонаправленных вектора скорости (повёрнутые относительно исходного на ±30–45° и с увеличенным модулем).

### Не-ответственности

- Не решает, когда астероид уничтожен — это делает `CollisionSystem` + `World` через выставление `alive = false`.
- Не создаёт стартовый набор астероидов в волне — это зона `WaveManager`; `Asteroid` только умеет быть созданным.
- Не начисляет очки за своё уничтожение — `Scoring` сам смотрит в `ASTEROID.POINTS[size]` по событию коллизии. `Asteroid` хранит `size`, но не начисляет.
- Не управляет списком «осколков»: `split()` возвращает массив, а вставку в `World.asteroids` и удаление исходного делает вызывающая сторона.
- Не знает про коллизии: использует `radius` из базового класса, саму проверку выполняет `CollisionSystem`.
- Не использует Canvas-трансформации (`ctx.translate`/`ctx.rotate`) — поворот применяется к вершинам аналитически; это упрощает отладку и не требует сохранять/восстанавливать состояние контекста.
- Не спавнит частицы взрыва — эффект создаёт `World` по событию коллизии.

## Публичный интерфейс

Единственный экспорт — класс:

- `class Asteroid extends Entity` — конкретная игровая сущность «астероид».

Публичные/инстансные поля:

- `size: 'large' | 'medium' | 'small'` — размер, определяет базовый радиус, скорость, очки и поведение `split()`.
- `shape: readonly number[]` — массив радиусов по равномерно распределённым углам (длина в диапазоне `[ASTEROID.VERTICES_MIN, ASTEROID.VERTICES_MAX]`). Генерируется один раз в конструкторе и после этого не меняется.
- `rotation: number` — текущий угол поворота полигона, радианы.
- `angularVelocity: number` — скорость вращения, рад/с; выбирается случайно в `[-ASTEROID.ANGULAR_SPEED_MAX, +ASTEROID.ANGULAR_SPEED_MAX]` в конструкторе.

Унаследованные от `Entity`: `position`, `velocity`, `radius`, `alive`.

Конструктор:

- `constructor(size: 'large' | 'medium' | 'small', position: Vec2, velocity: Vec2)` — задаёт размер и начальное движение; остальные параметры (`radius`, `shape`, `rotation`, `angularVelocity`) выводятся внутри из `config.ASTEROID` и генераторов случайных чисел.

Методы:

- `update(dt: number): void` — применяет `rotation += angularVelocity * dt`, затем `this.integrate(dt)` для движения с wrap-around.
- `draw(ctx: CanvasRenderingContext2D): void` — рисует замкнутую ломаную по `shape` с учётом `rotation` и `position`; обводка белая, без заливки.
- `split(): Asteroid[]` — возвращает массив из 0, 2 астероидов меньшего размера; см. раздел «Ключевые потоки».

## Модель данных

Модуль — один класс в памяти, без внешнего хранилища.

**Поля `Asteroid` (в дополнение к `Entity`):**

| Поле | Тип | Значение по умолчанию | Назначение |
|---|---|---|---|
| `size` | `'large' \| 'medium' \| 'small'` | из конструктора | категория размера |
| `shape` | `readonly number[]` | генерируется в конструкторе | радиусы вершин полигона для «шершавого» силуэта |
| `rotation` | `number` | `0` | текущий угол поворота, радианы |
| `angularVelocity` | `number` | `randomRange(-ANGULAR_SPEED_MAX, +ANGULAR_SPEED_MAX)` | скорость вращения, рад/с |

Дополнительно через базовый конструктор выставляются:

- `radius = config.ASTEROID.RADIUS[size]`
- `position`, `velocity` — из аргументов
- `alive = true`

**Форма `shape`.** Длина — случайное целое в `[VERTICES_MIN, VERTICES_MAX]`. Каждая вершина `i` — это радиус на угле `(i / N) * 2π`, равный `radius * (1 + randomRange(-ROUGHNESS, +ROUGHNESS))`. Таким образом массив — это полярное представление полигона с фиксированными углами и зашумлёнными радиусами. Один раз сгенерировав, его не меняем — это даёт стабильный узнаваемый силуэт конкретного астероида на всём времени его жизни.

Связей с другими сущностями и индексов нет — `Asteroid` живёт в плоском списке `World.asteroids`.

## Ключевые потоки

**Создание астероида.** `WaveManager.startWave(n)` или `Asteroid.split()` вызывают `new Asteroid(size, position, velocity)`. Конструктор: (1) берёт `radius = ASTEROID.RADIUS[size]`, (2) вызывает `super(position, velocity, radius)` для инициализации базовых полей, (3) определяет число вершин `N = randomInt(VERTICES_MIN, VERTICES_MAX)`, (4) заполняет `shape` длиной `N` радиусами вида `radius * (1 + randomRange(-ROUGHNESS, +ROUGHNESS))`, (5) задаёт `rotation = 0` и `angularVelocity = randomRange(-ANGULAR_SPEED_MAX, +ANGULAR_SPEED_MAX)`. После выхода из конструктора астероид полностью готов к тикам и отрисовке.

**Тик симуляции `update(dt)`.** Сначала применяется собственная физика вращения: `this.rotation += this.angularVelocity * dt`. Затем `this.integrate(dt)` из `Entity` сдвигает `position` на `velocity * dt` и оборачивает координаты через `wrapVec2(position, CANVAS.WIDTH, CANVAS.HEIGHT)`. `rotation` не нормализуется в `[0, 2π)` — значение растёт без ограничений, `Math.cos`/`Math.sin` корректно работают с любыми входами.

**Отрисовка `draw(ctx)`.** Для каждой вершины `i` вычисляется угол `θ = (i / N) * 2π + rotation` и радиус `r = shape[i]`; точка в мировых координатах — `position + {cos(θ) * r, sin(θ) * r}`. Все точки соединяются через `ctx.beginPath()` → `moveTo` / `lineTo` в цикле → `closePath()` → `ctx.strokeStyle = '#fff'` → `ctx.lineWidth = 1` → `ctx.stroke()`. Никаких `translate`/`rotate` у контекста не делается — поворот уже учтён в аналитических координатах вершин. Wrap-around отрисовка на краях экрана — ответственность утилит `Renderer` (рисуется дубль со смещением при необходимости), сам `Asteroid` рисует свой полигон один раз.

**Дробление `split()`.** Правила:

- `size === 'small'` → возвращается пустой массив `[]`. Вызывающий (World) просто удалит исходный астероид; эффект взрыва и очки — вне ответственности этого метода.
- `size === 'large'` → два `new Asteroid('medium', position, velocityA/B)`.
- `size === 'medium'` → два `new Asteroid('small', position, velocityA/B)`.

Векторы скорости осколков строятся так: берётся исходная `this.velocity`, поворачивается с помощью `Vec2.rotate` на случайный угол из диапазона `±[30°, 45°]` (конкретно: один осколок — на `+α`, другой — на `-β`, где `α, β ∈ [30°, 45°]`, градусы → радианы через `degToRad`), и её модуль увеличивается (например, умножением на коэффициент из диапазона `[1.2, 1.6]` — стандартное поведение Asteroids, чтобы осколки были «резвее» родителя; конкретное число — открытый вопрос тюнинга). Позиция обоих осколков — `this.position` исходного астероида; они мгновенно разъедутся за счёт разнонаправленных скоростей. `alive` у исходного `this.split()` не меняет — это делает вызывающая сторона.

**Взаимодействие с `World` при уничтожении.** `CollisionSystem.detect` возвращает событие `bulletAsteroid`; `World` по событию: (1) выставляет `bullet.alive = false` и `asteroid.alive = false`, (2) вызывает `asteroid.split()` и добавляет полученные осколки в `world.asteroids`, (3) начисляет очки через `Scoring.addPoints` по `asteroid.size`, (4) спавнит частицы взрыва. Следующий проход `world.asteroids = world.asteroids.filter(a => a.alive)` убирает уничтоженный исходник; новые осколки — уже живые, остаются.

## Зависимости

- **`entity`** — базовый класс `Entity`, его поля и `integrate(dt)`. `Asteroid.update` обязательно вызывает `super.integrate`.
- **`vec2-math`** — тип `Vec2`, функции `rotate` (для поворота векторов скоростей осколков в `split()`), `scale` (для увеличения модуля скорости осколков), `randomRange`, `randomInt`, `degToRad`. Также косвенно через `Entity.integrate` — `wrapVec2`.
- **`config`** — объект `ASTEROID`: `RADIUS[size]`, `VERTICES_MIN/MAX`, `ROUGHNESS`, `ANGULAR_SPEED_MAX`, `SPEED_MIN/MAX` (для выбора скорости при создании из `WaveManager`, не внутри конструктора — конструктор скорость получает аргументом), и размеры `CANVAS` (опосредованно через `integrate`).
- **Стандартный DOM (`CanvasRenderingContext2D`)** — только как тип параметра `draw(ctx)`.

Обратные зависимости: `World` (хранит список), `CollisionSystem` (использует как операнд событий), `WaveManager` (создаёт), `Scoring` (читает `size` из события коллизии).

## Обработка ошибок

- **Некорректный `size` в конструкторе.** Значение ограничено типом `'large' | 'medium' | 'small'`; компилятор не даст передать другое. Рантайм-проверок нет. В индексации `config.ASTEROID.RADIUS[size]` это гарантированно даст число.
- **`split()` у `small`.** Не ошибка, штатное поведение: возвращается пустой массив. Обработка — в `World`: просто удалить исходный астероид по флагу `alive` и не добавлять новых.
- **Downstream failure.** Модуль синхронный, без I/O и сети — понятие «downstream» неприменимо. Единственный внешний вызов — `ctx.stroke()` в `draw`; исключения из Canvas API (например, из-за порченого контекста) пробрасываются до верхнеуровневого try/catch в игровом цикле.
- **Partial success.** Неприменимо: `update`, `draw`, `split` либо отрабатывают полностью, либо выбрасывают исключение.
- **`NaN` в `velocity` / `position`.** Сам `Asteroid` не проверяет — как и `Entity`, он прозрачно пропускает значения через себя. Источником `NaN` может быть баг в `split()` (`normalize` нулевого вектора), но `Vec2.normalize` по контракту возвращает `ZERO` для нулевого входа, так что этот путь безопасен; поворот через `rotate` тоже `NaN`-safe для любых конечных чисел.
- **Invalid input (пустой `shape`).** Исключено по построению: `randomInt(VERTICES_MIN, VERTICES_MAX)` при `VERTICES_MIN ≥ 3` всегда даёт ≥ 3 вершин; конфиг гарантирует `VERTICES_MIN = 8`.

Наружу ошибок модуль не выбрасывает самостоятельно.

## Стек и библиотеки

- **TypeScript, класс ES2022 с `extends`.** Язык и парадигма заданы архитектурой (классический ООП, общий предок `Entity`). Никаких декораторов, миксинов, фабрик — обычный класс с конструктором и тремя методами.
- **Без внешних библиотек.** Ни геометрических (`polygon`, `clipper`), ни генераторов шума (`simplex-noise`): форма астероида достаточно простая, чтобы собрать её из `Math.random` и базовой тригонометрии.
- **Canvas 2D API — только базовые пути** (`beginPath`, `moveTo`, `lineTo`, `closePath`, `stroke`). `ctx.translate`/`rotate`/`save`/`restore` сознательно не используются — поворот вычисляется аналитически на координатах вершин. Причина: один проход рисования, без побочных эффектов на состоянии контекста, проще отлаживать.
- **Без пула объектов.** `split()` создаёт до двух новых экземпляров; на десятках астероидов за кадр GC-давление пренебрежимо.
- **Случайность — `Math.random` через утилиты `vec2-math`.** Отдельного PRNG с сидом нет (согласуется с решением `vec2-math`).

## Конфигурация

Собственных env-переменных и секретов модуль не имеет. Все числовые параметры читаются из `config.ASTEROID`:

| Имя | Назначение | Источник значения по умолчанию |
|---|---|---|
| `ASTEROID.RADIUS[size]` | базовый радиус коллизии и опорный радиус формы | `config.ts` (`40 / 20 / 10`) |
| `ASTEROID.VERTICES_MIN` / `VERTICES_MAX` | диапазон числа вершин полигона | `config.ts` (`8 / 12`) |
| `ASTEROID.ROUGHNESS` | амплитуда шума радиуса вершин (доля от базового) | `config.ts` (`0.35`) |
| `ASTEROID.ANGULAR_SPEED_MAX` | верхняя граница модуля угловой скорости | `config.ts` (`1.5` рад/с) |
| `ASTEROID.SPEED_MIN` / `SPEED_MAX` | диапазон модуля линейной скорости (используется при создании извне — `WaveManager`) | `config.ts` (`30 / 90` px/s) |

Никаких собственных внутренних констант класс не заводит.

## Открытые вопросы

- Точный коэффициент ускорения скорости осколков в `split()` (сейчас «`[1.2, 1.6]`») — подлежит тюнингу после первой играбельной сборки и, возможно, должен переехать в `config.ASTEROID` как `SPLIT_SPEED_MULTIPLIER_MIN/MAX`.
- Точный диапазон угла поворота скоростей осколков (`±[30°, 45°]`) — туда же, в `config.ASTEROID` как `SPLIT_ANGLE_MIN/MAX_DEG`, чтобы не прятать в коде класса.
- Нужно ли сохранять «унаследованный» поворот у осколков — то есть передавать в новый `Asteroid` исходные `rotation` и/или `angularVelocity` родителя, чтобы визуально осколки «продолжали» вращение. Пока генерируем независимые — проще и визуально достаточно.
- Стоит ли перенести поля `rotation` и `angularVelocity` в базовый `Entity` (там же есть открытый вопрос об этом) — если окажется, что `Ship` и `Ufo` тоже ими оперируют единообразно. Пока держим в `Asteroid`.
- Формат `shape`: `readonly number[]` (радиусы по фиксированным углам) против `readonly Vec2[]` (уже предвычисленные локальные точки). Второй вариант экономит один `cos`/`sin` на вершину за кадр, но требует повторного вычисления вершин при повороте — на 60 Hz разница пренебрежима; выбран первый как более компактный и явный.
- Кэшировать ли построенный `Path2D` вместо пересчёта в `draw` на каждом кадре. С учётом поворота кэш перестраивать придётся постоянно, так что выгоды нет — решение оставить ломаную «в лоб».
