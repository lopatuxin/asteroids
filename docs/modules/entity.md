# Модуль `entity`

## Назначение

Модуль определяет базовый абстрактный класс `Entity` — единый скелет и контракт для всех игровых объектов на сцене (`Ship`, `Asteroid`, `Bullet`, `Ufo`, `Particle`). Он фиксирует общие поля (позиция, скорость, радиус, флаг «живости») и требует от подклассов реализации двух методов — `update(dt)` и `draw(ctx)`, — что позволяет `World` хранить все сущности в плоских списках и прогонять их одним однообразным циклом. Без этого общего контракта каждая сущность пришлось бы обслуживать индивидуальным кодом, а цикл `update/draw`/фильтрация мёртвых объектов расползлись бы по сцене.

## Ответственности

- Объявление абстрактного класса `Entity` — единого предка всех игровых объектов.
- Хранение общего состояния любой сущности: `position: Vec2`, `velocity: Vec2`, `radius: number`, `alive: boolean`.
- Объявление обязательного контракта `update(dt): void` и `draw(ctx): void` — как `abstract`-методов, обязательных к реализации в подклассах.
- Предоставление защищённого helper-метода `integrate(dt)`, реализующего стандартный шаг «движение по инерции + wrap-around по размеру канваса». Подклассы вызывают его из собственного `update(dt)` после того, как применили свою специфику (тяга, вращение, изменение скорости).
- Инициализация полей через общий конструктор, принимающий `position`, `velocity`, `radius`; флаг `alive` по умолчанию равен `true`.

### Не-ответственности

- Не содержит конкретных сущностей (`Ship`, `Asteroid`, `Bullet`, `Ufo`, `Particle`) — каждая живёт в своём модуле.
- Не отвечает за коллизии — это зона `CollisionSystem`; `Entity` лишь предоставляет ей `position` и `radius` для расчётов.
- Не управляет списками сущностей и не удаляет мёртвые объекты — это делает `World`, фильтруя по флагу `alive`.
- Не знает о сценах, игровом цикле, вводе, волнах и очках.
- Не содержит специфики рендера (Canvas-примитивы, стили линий) — подпись `draw(ctx)` абстрактна, реализацию дают подклассы через утилиты `Renderer`.
- Не выполняет рантайм-валидацию аргументов конструктора (отрицательный радиус, `NaN`-координаты) — это контрактные нарушения, ловятся на уровне вызывающего кода.
- Не возвращает сущности обратно в «живое» состояние: переход `alive: true → false` односторонний.

## Публичный интерфейс

Единственный экспорт — абстрактный класс:

- `abstract class Entity` — базовый скелет игрового объекта.

Поля экземпляра:

- `position: Vec2` — текущая позиция центра сущности в координатах канваса.
- `velocity: Vec2` — текущий вектор скорости в px/s.
- `radius: number` — радиус описывающей окружности для коллизий и приблизительной отрисовки.
- `alive: boolean` — признак «жив/удалить»; `World` чистит списки фильтром по этому флагу.

Конструктор:

- `constructor(position: Vec2, velocity: Vec2, radius: number)` — задаёт стартовое состояние; `alive` инициализируется в `true`.

Абстрактные методы (обязательны к реализации в подклассах):

- `abstract update(dt: number): void` — продвигает симуляцию сущности на шаг `dt` секунд.
- `abstract draw(ctx: CanvasRenderingContext2D): void` — отрисовывает сущность на переданном контексте.

Защищённый helper:

- `protected integrate(dt: number): void` — единый шаг движения: `position = wrapVec2(position + velocity * dt, CANVAS.WIDTH, CANVAS.HEIGHT)`. Подклассы вызывают его из своих `update(dt)` после применения собственной физики (тяга, вращение, уменьшение TTL и т.п.).

## Модель данных

Модуль не владеет таблицами или коллекциями в смысле БД; он описывает форму одного объекта в памяти.

**Поля `Entity`:**

| Поле | Тип | Значение по умолчанию | Назначение |
|---|---|---|---|
| `position` | `Vec2` | из конструктора | центр сущности на канвасе |
| `velocity` | `Vec2` | из конструктора | скорость в px/s |
| `radius` | `number` | из конструктора | радиус коллизии и габарит отрисовки |
| `alive` | `boolean` | `true` | `false` → сущность будет удалена `World` на ближайшей зачистке |

**Связи и индексы.** Никаких — `Entity` не хранит ссылок на другие сущности и не участвует в индексных структурах. Подклассы добавляют собственные поля (см. модель данных в `architecture.md`: `heading`, `cooldown`, `size`, `lifetime` и т.п.).

**Выбор «абстрактный класс, а не интерфейс».** Интерфейс TypeScript описал бы форму (`position`, `velocity`, …, `update`, `draw`), но:

- Не даёт общей реализации `integrate(dt)` — её пришлось бы копировать в каждый подкласс или выносить в свободную функцию, теряя удобство `this.integrate(dt)`.
- Не гарантирует единообразной инициализации полей через общий конструктор — каждый подкласс описывал бы собственные поля отдельно, с риском разойтись в именах/типах.
- Не даёт дефолта `alive = true` в одном месте.

Абстрактный класс решает все три пункта: хранит состояние, предоставляет общий конструктор, даёт реализацию `integrate`, и при этом через `abstract`-методы всё равно обязывает подклассы реализовать `update` и `draw`.

## Ключевые потоки

**Шаг симуляции подкласса через `integrate`.** Подкласс (например, `Asteroid`) в своём `update(dt)` сначала применяет свою специфику — у астероида это `this.rotation += this.angularVelocity * dt` (у пули — `this.lifetime -= dt` и проверка `if (this.lifetime <= 0) this.alive = false`). Затем вызывает `this.integrate(dt)`: helper сдвигает `position` на `velocity * dt` и оборачивает координаты через `wrapVec2(position, CANVAS.WIDTH, CANVAS.HEIGHT)`. Итог — сущность корректно движется по экрану-тору без дублирования кода wrap в каждом подклассе.

**Смерть сущности.** Когда условие «мёртв» срабатывает (истёк `lifetime` пули, пуля попала в астероид, корабль столкнулся с НЛО и уже без неуязвимости), ответственный код (сам подкласс в `update(dt)` либо `World` при разрешении коллизий) просто выставляет `this.alive = false`. После обработки всех коллизий `World` прогоняет списки фильтром `list.filter(e => e.alive)` и убирает мёртвые объекты. Память освобождает GC на следующем цикле.

**Создание новой сущности взамен старой.** Поскольку `alive` обратно в `true` не возвращается, «воскрешение» корабля после смерти — это создание нового экземпляра `Ship(...)` с начальной позицией в центре и таймером неуязвимости, а не сброс флага на старом объекте. Это исключает класс багов, где «мёртвая» сущность случайно остаётся в списке и продолжает обрабатываться после того, как её уже учли в очках/жизнях.

**Отрисовка.** `GameScene.draw(ctx)` перебирает все списки (`asteroids`, `bullets`, `ufos`, `particles`, `ship`) и вызывает `entity.draw(ctx)` для каждого. Диспетчеризация — полиморфная: `Entity.draw` абстрактен, вызовется конкретная реализация подкласса, использующая утилиты `Renderer` для векторных линий.

## Зависимости

- **`vec2-math`** — типы и операции над векторами: `Vec2`, `add`, `scale`, `wrapVec2`. Используется в `integrate(dt)` и в конструкторе (типизация полей).
- **`config`** — размеры канваса `CANVAS.WIDTH` и `CANVAS.HEIGHT`, нужные `wrapVec2` в `integrate`.
- **Стандартный DOM (`CanvasRenderingContext2D`)** — только как тип параметра `draw(ctx)`; сам модуль рендером не занимается.

Обратные зависимости (кто импортирует `Entity`): все модули сущностей — `Ship`, `Asteroid`, `Bullet`, `Ufo`, `Particle`. Также `World` и `CollisionSystem` могут использовать тип `Entity` для общих списков и пар столкновений (`CollisionEvent { a: Entity, b: Entity, … }`).

## Обработка ошибок

- **Некорректный вход в конструктор (`radius < 0`, `NaN`-координаты).** Это контрактное нарушение со стороны подкласса — проверок нет, значение принимается как есть. Отрицательный радиус проявится сразу: коллизии перестанут срабатывать, отрисовка даст видимый артефакт — баг будет виден и пойман в dev-сборке.
- **Попытка «воскресить» сущность присваиванием `alive = true`.** Типом не запрещено, но контрактом — запрещено: подразумевается односторонний переход `true → false`. Нарушение контракта может привести к тому, что удалённая сущность окажется в списках после того, как её уже отработали в коллизиях. Защита — дисциплина вызывающего кода: «новая сущность = новый экземпляр», а не мутация флага.
- **Исключение в `update(dt)` или `draw(ctx)` подкласса.** `Entity` ничего не ловит. Исключение пробрасывается вверх до верхнеуровневого try/catch в игровом цикле (см. `architecture.md`, раздел «Обработка ошибок»): в dev-сборке игра остановится с сообщением, в проде — вернётся в `MenuScene`.
- **Downstream failure, partial success.** Неприменимо: модуль синхронный, без I/O и без внешних вызовов. `integrate(dt)` не может «частично получиться» — либо отработал, либо выбросил исключение на арифметике (что означает `NaN` в `position`/`velocity`, то есть баг вызывающего, не этого модуля).

Наружу ошибок модуль не выбрасывает самостоятельно: исключений не кидает, только возвращает значения или мутирует собственные поля.

## Стек и библиотеки

- **TypeScript (ES2022 target), классы с `abstract`.** Язык задан архитектурой; `abstract class` и `abstract`-методы появились в TS достаточно давно и компилируются в обычный ES2022-класс с проверкой «нельзя инстанцировать» на уровне типов. Это единственно необходимая фича языка для модуля.
- **Без внешних библиотек.** Модуль — десяток строк поверх `vec2-math` и `config`; зависимости типа `mitt`, `rxjs` или ECS-фреймворка не требуются.
- **Без рантайм-проверок.** Никаких `if (radius < 0) throw` и подобного: контракты обеспечивает компилятор TS, а семантические нарушения (отрицательный радиус, `alive` обратно в `true`) ловятся визуально в dev-сборке.
- **Без пула объектов.** Каждый подкласс создаётся обычным `new` и удаляется через `alive = false` + фильтр списка; GC-давление на десятках сущностей при 60 Hz пренебрежимо.

## Конфигурация

Внешней конфигурации модуль не имеет — ни env-переменных, ни секретов, ни рантайм-настроек. Единственное, что он читает извне при работе — константы `CANVAS.WIDTH` и `CANVAS.HEIGHT` из модуля `config`, и делает это в `integrate(dt)` через обычный `import`. Изменение этих значений — правка `config.ts` и пересборка бандла.

Внутренних констант модуль не заводит.

## Открытые вопросы

- Делать ли `draw(ctx)` опциональным и предоставлять в `Entity` пустую дефолт-реализацию — например, `Particle` рисуется одной точкой/линией и может захотеть обойтись без собственного метода. Пока оставляем строгим `abstract`, пересмотрим при реализации `Particle`.
- Нужен ли общий метод `kill(): void` как синтаксический сахар поверх `this.alive = false` (плюс потенциальное место для эмита события «entity died» для частиц взрыва). Решение отложено до реализации `Particle` и эффектов взрыва.
- Вводить ли в `Entity` обобщённый `angularVelocity` и `rotation` — ими пользуются `Asteroid` и косвенно `Ship` (через `heading`). Пока держим в подклассах, чтобы не тянуть неиспользуемое поле в `Bullet`/`Particle`; пересмотрим, если обнаружится дубль кода вращения.
- Стоит ли параметризовать `integrate(dt)` флагом «wrap / не wrap» на случай, если какая-то будущая сущность (например, заэкранный спавнер) не должна wrap-иться. На сегодня все сущности из архитектуры wrap-ятся одинаково — флаг не нужен.
